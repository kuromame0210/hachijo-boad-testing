プロジェクト『Allo』総合開発仕様書 (Master Specification)


1. プロジェクト・ビジョン (Vision)
コンセプト：八丈島の「生活OS」
単なる掲示板ではなく、島民が毎朝必ず開く「生活インフラ（交通・天気・ゴミ）」を入り口とし、そこから「物流（お裾分け・売買）」、「仕事」、「防災」へと繋がる地域密着型プラットフォーム（PWA）。

ビジネスゴール
島内No.1メディアの確立: 全島民の生活導線を握り、広告媒体としての価値を高める。

studioBAUM経済圏の拡大: アプリを通じて自社の本業（映像制作・ドローン・音楽教室）への自然な送客・受注システムを構築する。

地域課題の解決: 物流コスト削減、フードロス解消、災害時情報の集約、サスティナブル。

2. デザイン・UI規定 (Visual Identity)
2.1 世界観・カラーパレット
ブラウザアプリ「Dia」のような、先進的かつ自然なグラデーションを採用。

Hachijo Gradient: 八丈島の1日を表現するグラデーションをヘッダーや背景に適用。

Top: Hachijo Blue (爽やかな青・空)

Middle: Island Sun (暖かな黄色・光)

Bottom: Sunset Red (夕焼け・情熱)

Typhoon Mode (緊急時): 上記グラデーションを廃止し、**「黒・赤・黄」**のハイコントラストな警告色へ切り替える。

2.2 UIレイアウト (Mobile First)
App Shell: 画面下部のボトムナビゲーション (Dashboard, Post, View, MyPage)。

Typography: 視認性重視。入力フォームは極力排除し、タップ（タグ選択）中心のUX。

3. システム・アーキテクチャ (Technical Stack)
Frontend: Next.js (React), Tailwind CSS

Platform: PWA (Progressive Web App) - iOS/Androidのホーム画面追加を推奨

Backend: Firebase (Authentication, Firestore, Storage, Cloud Functions)

Scraping: Puppeteer / Cheerio (Cloud Functions上で定期実行しFirestoreへキャッシュ)

Map: Google Maps JavaScript API

Infrastructure: Vercel (Frontend), Firebase (Backend)

重要ロジック：GPSジオフェンシング
ユーザーの現在地（緯度経度）に基づき、権限を動的に変更する。

4. 機能詳細要件 (Functional Requirements)
【Dashboard】トップ・ダッシュボード
島民が「朝起きて1秒」でその日の行動を決定できるコックピット画面。

A. 交通運行ステータス (Traffic Lights UI)
スクレイピングにより以下の情報を取得し、「信号機」形式で表示。

ANA: 羽田発 1便・2便・3便 それぞれに信号を表示。

橘丸 (東海汽船): 大型客船の就航状況。

青ヶ島丸: 橘丸の下または横に配置。

ヘリ (愛ランドシャトル): 運行状況。

ステータス色: 🔵就航/定刻, 🟡条件付/遅延, 🔴欠航, ⚪未定

B. ライフラインパネル
Weather: 天気アイコン、気温。

Ocean: 波の高さ (Wave Height) ※漁師・釣り人・ダイバー向け最重要情報。

Living:

ゴミ出しナビ: ユーザー設定地区（5地区）の「今日のゴミ」をアイコン表示（🏠アイコン）。

島ことば: 日替わりで島の方言を1つ表示。

C. 広告バナー (Ad Space)
studioBAUM自社広告およびスポンサー枠。

【Marketplace】投稿・カテゴリ仕様
A. カテゴリと自動削除ルール
情報の鮮度を保つため、厳格な自動削除ルールを設ける。

🐟 生鮮・お裾分け (Fresh)

削除: 「日没」 または 24時間後。

機能: 「大漁フラグ」（釣り人が「釣れた！」ボタンを押すと即時プッシュ通知）。

🛋 不用品・譲ります (Goods)

削除: 7日間。

禁止: 車両（車・バイク）の売買（名義変更トラブル防止）。

📢 お店・求人 (Biz)

削除: 「当日の日没1時間前（ランチ等）」または「7日間（求人）」を選択式。

B. 取引・チャット (Direct Message)
プライバシー保護のため、コメント欄ではなく1対1のDMを採用。

仕様: テキストメッセージのみ。

機能制限: 既読機能なし（トラブル防止）、通報・ブロック機能必須。

用途: 受け渡し場所や時間の調整。

【My Page】アカウント・設定
ユーザー共通:

居住地区設定（三根/大賀郷/樫立/中之郷/末吉） ※ゴミ出しロジックに必須。

事業者アカウント (Business Account):

自社Webサイトへのリンク。

フォトギャラリー（最大30枚）。

自社情報（営業時間など）。

【Typhoon Mode】防災・緊急モード
トリガー: 管理者手動 または 気象庁API（暴風警報）。

UI変更: グラデーションを廃止し、**「黒背景 × 赤文字 × 黄色」**の警告色へ強制切り替え。

優先表示:

Windy等の進路予想図。

停電情報（TEPCO）。

スーパー・GSの営業状況。


ターゲット 課題 (Pain) Allo提供価値 (Gain)
農家 規格外野菜の廃棄 写真を撮るだけで近所の人が取りに来る（フードロスゼロ）。
釣り人 釣れすぎた魚の処理 「大漁フラグ」で帰港前に貰い手・買い手が決まる。
飲食店 雨の日の客足減 「雨の日限定サービス」等のリアルタイム集客。
一般島民 船・ゴミ確認の手間 毎朝のルーティンが1秒で完結。送料シェアで物資調達。
観光客 欠航時の不安 正確な交通情報と、ローカルな島体験の提供。
八丈町 防災無線が届かない 予算ゼロで全町民へ届く「デジタル回覧板」。
弊社(BAUM) 自社認知・営業 島内No.1メディアのオーナーとなり、クリエイティブ案件を受注。



6. ロードマップ (Release Schedule)
Phase 0 (Dev): allo.tokyo ドメイン設定、プロトタイプ開発。

Phase 1 (Release):

機能制限版リリース（交通・天気・ゴミ出しのみ）。

目的：島民のスマホに「便利ツール」としてインストールさせる（習慣化）。

Phase 2 (Growth):

「0円譲渡」カテゴリの解禁。

目的：お金の絡まない取引で、アプリ内の流動と信頼を作る。

まずは、顔のわかる事業者から平田が直接声をかけて行って30者程度を巻き込み、
「alloはお得情報が集まってる！」&
交通情報あるからついつい開いちゃう！の習慣化を行う。

Phase 2.5
八丈町を後援につける
＜広報八丈＞を閲覧できるようにする。
地域DXを無償で備えられる八丈町は得しかない。

Phase 3 (Monetization):

「売買（価格設定）」解禁。

事業者アカウント機能（個人間決済ができる簡単なチェック項目をクリアしたら許可）、広告枠の販売開始。

Phase 4 (Expansion):

防災・台風モードの高度化（自治体連携）。
八丈町災害系の予算から引っ張りたい期待。
ドローン配送等の実証実験連携。

申し送り事項
スクレイピングの堅牢性: 交通機関（特に東海汽船・ANA）のサイト構造変更に強い設計、またはエラー時のフォールバック（手動更新機能）を用意すること。
